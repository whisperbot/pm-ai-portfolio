# Insight 01｜AI 应用中，什么场景适合“小模型”方案

> 来源：内部项目方法沉淀（公开版已做脱敏与抽象）
> 
> 关键词：提速降本、任务难度分层、业务定制化、模型选型

## TL;DR（先看结论）
- 小模型不是“过渡方案”，而是 AI 工程化里的**长期选项**。
- 当你追求**时延/成本**、任务是**结构化简单任务**、且业务有较强**定制化需求**时，应优先评估小模型。
- 当任务依赖**复杂推理**、**强世界知识**、**高 PE（Prompt Engineering）可塑性**时，大模型通常更稳。

## 1. 为什么这个问题重要
AI 产品落地时，常见矛盾是：
- 大模型效果更强，但速度慢、成本高、定制成本高；
- 小模型更轻、更快、更便宜，但能力边界需要更精细管理。

选型不是“谁更先进”，而是“谁在这个场景下 ROI 更高”。

## 2. 何时优先考虑小模型

### 2.1 对时延/成本敏感
- 实时交互、高并发、成本敏感场景，优先考虑小模型。
- 典型价值：响应更快、单次调用更低、可更大规模部署。

### 2.2 任务难度相对简单
- 意图识别、信息抽取、文本分类、结构化映射等任务。
- 实操判断法：**标注标准能快速对齐的任务，通常更适合小模型。**

### 2.3 业务定制性强
- 场景规则多、术语重、需要频繁微调迭代。
- 小模型 SFT 成本更低、迭代更敏捷，适合做垂类“专才”。

## 3. 哪些场景不建议直接用小模型

### 3.1 强逻辑推理任务
- 复杂跨步推理、隐含意图推断、复杂时间推理。

### 3.2 高度依赖世界知识的任务
- 尤其是需要广泛常识和领域知识联动时。
- 无论大/小模型，涉及知识时效性都建议接 RAG。

### 3.3 强 PE 驱动的动态规则场景
- 规则变化快、提示词要求复杂且持续演化。
- 模型越小，PE 规则遵循通常越弱。

### 3.4 严格时间标准化输出
- 例如要求精确输出标准时间格式并处理相对时间推算。
- 常见做法：小模型先做轻归一化，再交给专用解析工具处理。

## 4. 实战选型框架（可直接复用）

从 4 个维度打分（1~5 分）：
- **时延/成本压力**
- **任务复杂度（推理深度）**
- **知识依赖强度**
- **业务定制化强度**

建议决策：
- 高时延压力 + 低复杂度 + 高定制化 → **小模型优先**
- 高推理/高知识依赖 + 规则高动态 → **大模型优先**
- 混合场景 → **大小模型协同 + 工具链/RAG 兜底**

## 5. 产品落地建议
1. 不要用“模型大小”做一刀切决策，用“任务拆分”做决策。
2. 把复杂任务拆成“可由小模型完成的子任务 + 工具处理链路”。
3. 先跑小规模实验：观察质量、时延、成本三角平衡。
4. 选型结论要绑定指标：准确率、时延、成本、稳定性。

## 6. 关键认知
- 小模型的价值不在“替代大模型”，而在“提升系统整体性价比”。
- 真正可持续的方案，是能随着模型进化持续获益的方案。
