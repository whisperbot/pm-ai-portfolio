# Insight 02｜AI Music 小模型技术方案：LLM-lite 意图理解框架

> 来源：AI Music 项目中的小模型技术方案（公开版已做脱敏与抽象）
> 
> 关键词：Intent Routing、Function Call、低时延、海量工具召回、复杂意图规划

## TL;DR
- 在 AI 音乐等强交互场景里，**意图理解不是“锦上添花”，而是主链路基础设施**。
- 直接用大模型 + Prompt 做分流，常见问题是：慢、贵、稳定性不足、规模上复杂度爆炸。
- LLM-lite 的核心价值：用“小模型 FC 能力 + 前置召回 + 复杂意图规划”，在**质量、时延、可扩展性**之间找到工程最优解。

## 1. 问题定义
在用户 Query 到服务执行的链路里，关键一步是：
- 识别用户意图；
- 选择正确的服务（Agent/Workflow/Tool）；
- 提取调用参数；
- 无法匹配时要拒识。

如果这一步不稳，后面做得再好也会被“错分流”拖垮。

## 2. LLM-lite 三层能力设计

### 2.1 意图理解（Intent Understanding）
目标：从给定服务中选 Top1/TopK，并生成调用参数。

### 2.2 能力无限（Pre-recall for Massive Tools）
目标：当可用服务规模很大时，做前置召回，把候选集合先缩小给意图理解模块。

### 2.3 QueryEngine（复杂意图处理）
目标：识别复杂/多意图 Query，并做拆解与规划（支持 re-planning）。

这三层组合后，既能处理日常简单请求，也能兜住复杂请求，不把复杂度都压在一个模型上。

## 3. 两套 FC 落地路径

## 3.1 LLM-lite-FC（自回归）
适合：接入快、和大模型 FC 接口对齐、工程改造成本低。

任务定义：
- 输入 Query + 候选服务；
- 输出目标服务 + 参数 JSON；
- 不匹配则拒识。

工程边界（关键）：
- 只做单工具调用；
- 不在该模块内做多轮澄清；
- 参数归一化/后处理交给后链路。

## 3.2 LLM-lite-SAR-FC（半自回归）
适合：强时延场景（例如播控类）。

核心思路：
- 把“工具选择”和“参数抽取”拆开；
- 参数抽取按字段并行处理；
- 最后拼装 Function Call 结果。

价值：相比一次性长文本生成，可显著降低端到端时延。

## 4. 典型工程启发（对 PM 很有用）
1. **先拆任务，再选模型**：意图选择、参数抽取、复杂规划不要混成一坨。
2. **时延优先级要前置写进 PRD**：低时延场景应从架构层设计并行化。
3. **拒识能力是产品能力**：不乱答、不乱调工具，比“硬猜”更稳。
4. **干预机制必须产品化**：前置规则、意图结果干预、参数结果干预要可配置。

## 5. 可复用架构模板
```text
User Query
  -> (可选) 海量工具前置召回
  -> 意图理解（工具选择 + 参数抽取）
  -> 复杂意图识别（是否进入 Planner）
  -> 执行/编排
  -> 结果返回 + 监控评估
```

## 6. 适用场景建议
- 工具数量增长快、规则复杂的 AI 应用；
- 对时延敏感（如实时控制、对话链路前置路由）；
- 需要“高可控 + 可扩展 + 可持续迭代”的业务。

## 7. 一句话总结
LLM-lite 的意义不是“把模型做小”，而是把意图理解做成一套**可工程化、可扩展、可运营**的能力底座。
